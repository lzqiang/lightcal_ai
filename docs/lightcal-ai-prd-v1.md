# 轻卡 AI PRD v1

日期：2026-06-12  
版本：v1.0  
产品定位：面向普通减脂用户的 AI 饮食与运动记录 App

## 1. 产品目标

轻卡 AI 帮助用户用最少操作完成每日饮食和运动记录。用户通过拍照上传食物、上传运动截图或手动补录运动，系统使用 DeepSeek 大模型分析摄入与消耗，给出当天状态、昨日回顾、历史趋势和可执行的食谱与运动建议。

第一版不做专业医疗减重，也不追求克级营养精度。核心目标是降低记录门槛，让用户每天知道“今天吃得怎么样、动得够不够、明天怎么调”。

## 2. 目标用户

- 想减肥但不愿复杂记热量的人。
- 经常外卖、食堂或家常饭，难以精确称重的人。
- 愿意拍照记录食物和上传运动截图的人。
- 需要每天获得简单反馈和行动建议的人。

## 3. MVP 范围

### 3.1 必须包含

- 手机号或微信登录。
- 填写基础身体信息：性别、年龄、身高、当前体重、目标体重、减脂速度、饮食偏好、运动基础。
- 填写目标截止时间，用于判断目标是否合理并生成每日建议摄入范围。
- 首页展示今日摄入、今日消耗、今日状态。
- 首页展示昨日摄入、昨日消耗、昨日状态。
- 拍照或上传食物图片。
- DeepSeek 分析食物内容、份量和热量。
- 用户确认餐次和份量。
- 上传运动截图。
- DeepSeek 识别截图中的运动类型、时长、距离和热量。
- 手动补录运动。
- 历史记录列表，展示每天的摄入、消耗和状态。
- AI 每日总结。
- AI 推荐明日食谱。
- AI 推荐明日运动。
- 展示今日计划和明日计划。
- 展示计划日历，用周视图呈现每日摄入目标、运动安排和完成状态。

### 3.2 第一版不包含

- 社区、排行榜、打卡广场。
- 商城、私教、会员体系。
- 可穿戴设备同步。
- 复杂营养报表。
- 医疗诊断和疾病饮食建议。
- 家庭共享。

## 4. 核心用户流程

### 4.1 新用户首次使用

1. 打开 App。
2. 点击“开始记录”。
3. 如果未登录，弹出手机号或微信快捷登录。
4. 登录成功后，系统检查用户是否已填写基础信息。
5. 如果未填写，进入基础信息页。
6. 用户填写基础信息和目标截止时间。
7. 系统生成每日建议摄入范围。
8. 进入首页。

### 4.1.1 老用户启动路径

1. 打开 App。
2. 点击“开始记录”或等待自动进入。
3. 如果已登录且基础信息完整，直接进入首页。
4. 如果登录过期，先进入登录，再回到首页。

### 4.2 记录饮食

1. 用户点击“拍照记饮食”。
2. 拍摄或上传食物图片。
3. DeepSeek 识别食物和估算热量。
4. 用户选择餐次：早餐、午餐、晚餐、加餐。
5. 用户确认份量：少量、正常、偏多。
6. 保存记录。
7. 首页摄入数据更新。

### 4.3 记录运动

1. 用户点击“上传运动截图”。
2. 上传 Keep、Apple Fitness、微信运动、跑步机等截图。
3. DeepSeek 识别运动数据。
4. 如果截图无热量，系统根据运动类型、时长、强度估算。
5. 用户确认后保存。
6. 首页消耗数据更新。

### 4.4 查看反馈

1. 用户进入首页。
2. 查看今日状态和昨日回顾。
3. 进入历史记录查看过去天数。
4. 点击某一天查看当日饮食、运动和 AI 总结。

## 5. 页面设计

### 5.1 启动页

核心内容：
- 产品名：轻卡 AI。
- 标语：拍一下，AI 帮你管住热量。
- 主按钮：开始记录。

交互：
- 点击“开始记录”后，未登录用户进入登录页。
- 已登录但资料未填写完整的用户进入基础信息页。
- 已登录且资料完整的用户直接进入首页。

### 5.1.1 登录页

核心内容：
- 产品名和一句简短说明。
- 手机号输入框。
- 验证码输入框。
- 获取验证码按钮。
- 登录按钮。
- 微信登录按钮。
- 用户协议和隐私政策勾选。

交互：
- 用户必须勾选协议后才能登录。
- 手机号作为账号唯一标识，登录后在“我的”页脱敏展示。
- 手机号登录成功后，如果基础信息未填写，进入基础信息页。
- 手机号登录成功后，如果基础信息完整，进入首页。
- 微信登录成功后逻辑相同。
- 如果微信登录未绑定手机号，第一版要求补充手机号后再进入 App。
- 第一版不做复杂密码登录。

### 5.2 基础信息页

字段：
- 昵称。
- 性别。
- 年龄。
- 身高。
- 当前体重。
- 目标体重。
- 目标截止时间。
- 每周目标：轻松减、正常减、快速减。
- 饮食偏好：外卖党、食堂党、自己做饭、不吃辣、不吃猪肉、素食。
- 运动基础：几乎不运动、偶尔运动、经常运动。

输出：
- 每日建议摄入范围。
- 目标可行性提示，例如“目标偏快，建议延长 2 周”。
- 初始食谱和运动建议。

### 5.3 首页

首页是已登录且基础信息完整用户的默认进入页。它不是介绍页，而是每日执行面板。

顶部区域：
- 用户问候，例如“早上好，今天继续轻卡计划”。
- 连续记录天数。
- 当前目标进度，例如“距离目标还差 6.8 kg”。

核心模块：
- 今日进度：摄入、消耗、状态、剩余可摄入热量。
- 昨日回顾：昨日摄入、昨日消耗、昨日状态和 AI 简评，放在今日进度下方。
- 今日计划：早餐、午餐、晚餐、训练计划。
- 记录完成度：饮食已记录几餐、运动是否完成。
- 快捷操作：拍照记饮食、手动记饮食、上传运动截图。

交互：
- 点击今日进度，进入今天的单日详情页。
- 点击昨日回顾，进入昨天的单日详情页。
- 点击拍照记饮食，进入饮食识别页。
- 点击手动记饮食，进入手动饮食记录表单。
- 点击上传运动截图，进入运动识别页。

展示原则：
- 首页优先展示“今天还差什么”，而不是介绍产品能力。
- 今日计划必须放在首页可见区域，用户不用跳转到推荐页也能照着执行。
- 如果昨天状态为超标或略超标，首页提示应温和说明调整方向，不做负面评价。

### 5.3.1 我的页

我的页用于展示和管理账号资料、身体目标和基础设置。

顶部资料：
- 头像或默认头像。
- 昵称。
- 手机号，脱敏展示，例如 138****5678。
- 当前连续记录天数。

目标信息：
- 当前体重。
- 目标体重。
- 目标截止时间。
- 每日建议摄入范围。

功能入口：
- 编辑个人资料：昵称、头像、手机号绑定。
- 编辑身体信息：身高、体重、目标体重、目标截止时间。
- 饮食偏好。
- 运动基础。
- 用户协议。
- 隐私政策。
- 退出登录。

第一版规则：
- 手机号必须有。
- 昵称可选；如果用户未填写，默认显示“轻卡用户”加手机号后四位。
- 头像可选；如果用户未上传，使用默认头像。

### 5.4 饮食识别页

内容：
- 图片预览。
- AI 识别食物列表。
- 估算总热量。
- 识别置信提示。
- 餐次选择。
- 份量选择。
- 保存按钮。

手动记录：
- 支持用户不拍照，直接手动输入食物名称。
- 支持选择餐次、份量和估算热量。
- 手动记录保存后进入今日记录，并参与今日摄入统计。

### 5.4.1 手动饮食记录页

入口：
- 首页点击“手动记饮食”。
- 饮食识别失败时点击“手动输入”。

字段：
- 餐次：早餐、午餐、晚餐、加餐。
- 食物名称。
- 份量：少量、正常、偏多。
- 估算热量。
- 备注，可选。

交互：
- 用户输入食物名称后，可点击“AI 估算热量”。
- 用户也可以手动填写热量。
- 支持一餐添加多个食物。
- 保存后回到首页，今日摄入和今日记录更新。

第一版规则：
- 不要求用户填写克数。
- 份量以少量、正常、偏多为主。
- AI 估算结果必须允许用户修改。

提示文案：
- “AI 估算可能存在偏差，请确认份量后保存。”

### 5.5 运动识别页

内容：
- 截图预览。
- AI 提取运动类型、时长、距离、热量。
- 如果识别失败，展示手动输入入口。
- 保存按钮。

### 5.6 推荐页

模块：
- 明日计划总览。
- 明日三餐建议。
- 可替换食材。
- 今日补救运动。
- 明日推荐运动。
- 计划日历。
- 更新计划按钮。

推荐原则：
- 简单。
- 容易购买。
- 容易执行。
- 不鼓励极端节食。

### 5.6.1 计划日历

计划日历用于展示未来 7 天的减脂计划，不做复杂排课。

展示内容：
- 日期和星期。
- 每日建议摄入范围。
- 推荐运动类型和时长。
- 计划状态：未开始、进行中、已完成、需调整。

交互：
- 默认在推荐页展示横向 7 天日历。
- 点击某一天，展示当天食谱和运动计划。
- 如果用户当天实际摄入明显超出计划，次日计划由 AI 自动微调，但不做惩罚式极端控制。
- 用户点击“更新计划”后，系统基于昨天和最近历史记录重新生成未来计划。
- 更新计划应明确提示依据，例如“已根据昨天略超标和近 7 天运动不足调整”。

### 5.6.2 更新计划

入口：
- 推荐页“更新计划”按钮。

触发条件：
- 用户主动点击。
- 每天首次进入推荐页时，如果昨天数据已完整，可提示用户更新。

更新依据：
- 昨日摄入、消耗、状态。
- 最近 7 天摄入趋势。
- 最近 7 天运动完成情况。
- 当前目标体重和目标截止时间。

更新结果：
- 明日摄入目标。
- 明日三餐建议。
- 明日运动建议。
- 未来 7 天计划日历。

限制：
- 第一版不做自动频繁重算。
- 同一用户每天默认最多主动更新 3 次，避免成本失控。
- 更新后的计划不能建议极端节食或惩罚式运动。

### 5.7 历史记录页

列表字段：
- 日期。
- 摄入。
- 消耗。
- 状态。

点击某一天进入详情：
- 当天饮食列表。
- 当天运动列表。
- 摄入合计。
- 消耗合计。
- AI 总结。

### 5.8 单日详情页

入口：
- 历史记录页点击任意一天。
- 首页点击“昨日回顾”。

内容：
- 日期和状态。
- 当日摄入合计。
- 当日运动消耗合计。
- 当日净摄入：摄入减运动消耗。
- 饮食记录列表，按早餐、午餐、晚餐、加餐展示。
- 运动记录列表，展示运动类型、时长、消耗。
- AI 当日总结。
- AI 改进建议。

页面操作：
- 返回历史记录。

限制：
- 单日详情页为只读复盘页。
- 不提供编辑饮食记录和编辑运动记录入口。
- 如果后续需要修改记录，应从记录列表或独立记录管理入口处理，不放在详情复盘页。

## 6. 状态规则

系统根据每日建议摄入范围、实际摄入和运动消耗生成状态。

- 控制良好：摄入在建议范围内，且有基础运动或未明显超标。
- 略超标：摄入略高于建议上限，或运动不足。
- 超标：摄入明显高于建议上限，且运动消耗较低。
- 摄入偏低：摄入明显低于建议下限，提示避免极端节食。

状态文案应克制，不使用羞辱、恐吓或医疗诊断式表达。

## 7. DeepSeek 接入方案

### 7.1 模型用途

DeepSeek 用于四类任务：

1. 食物图片分析：识别食物、估算份量、估算热量。
2. 运动截图分析：识别运动类型、时长、距离、消耗热量。
3. 每日总结：分析当天摄入、消耗和主要问题。
4. 推荐生成：生成明日食谱和运动建议。

### 7.2 接入方式

服务端统一封装 DeepSeek API，不在客户端直连大模型。

客户端流程：
1. 上传图片到服务端。
2. 服务端做图片存储和安全检查。
3. 服务端调用 DeepSeek。
4. 服务端解析结构化 JSON。
5. 客户端展示结果并让用户确认。

### 7.3 输出格式要求

DeepSeek 返回必须结构化，示例：

```json
{
  "items": [
    {
      "name": "米饭",
      "portion": "正常",
      "calories": 280,
      "confidence": 0.82
    }
  ],
  "total_calories": 680,
  "summary": "主食偏多，蛋白质中等，蔬菜不足。",
  "warnings": ["图片角度可能影响份量估算"]
}
```

### 7.4 AI 安全边界

- 不提供医疗诊断。
- 不建议极端节食。
- 不建议未成年人快速减重。
- 不输出危险运动方案。
- 对识别不确定的内容明确提示用户确认。

## 9. 技术路线

### 9.1 客户端选择

第一版客户端采用 Flutter 跨平台开发，不采用 HTML5 WebView 套壳作为主架构。

选择 Flutter 的原因：
- 一套代码覆盖 iOS 和 Android，适合小团队快速迭代。
- UI 表现稳定，适合首页数据卡片、历史列表、AI 结果页、推荐页和后续趋势图。
- 拍照、相册、图片压缩、权限、推送、本地缓存、图表等能力都有成熟插件。
- 后续扩展体重趋势、饮食日历、AI 对话、订阅会员、可穿戴设备接入时，不需要从 H5 套壳迁移到原生。

HTML5 仅用于辅助页面：
- 用户协议。
- 隐私政策。
- 帮助中心。
- 运营活动页。

### 9.2 服务端选择

服务端采用 Java 技术栈，推荐 Spring Boot。

服务端职责：
- 用户登录和账号体系。
- 用户身体信息和目标管理。
- 饮食记录、运动记录、每日汇总。
- 图片上传、图片存储、图片安全检查。
- DeepSeek API 统一封装。
- AI 结果结构化解析、失败重试和降级。
- 食谱与运动推荐生成。
- 历史记录查询。

客户端不直接调用 DeepSeek，避免 API Key 泄露，也方便服务端统一控制成本、日志、重试和安全策略。

### 9.3 推荐系统架构

```text
Flutter App
  -> Java API 服务
    -> 文件存储
    -> 业务数据库
    -> DeepSeek AI 服务封装
    -> 每日汇总与推荐任务
```

第一版不需要复杂微服务。建议采用单体 Spring Boot 应用，内部按模块拆分：
- user：用户与身体信息。
- record：饮食和运动记录。
- summary：每日汇总和状态判断。
- ai：DeepSeek 调用、提示词、结构化解析。
- recommend：食谱和运动推荐。
- file：图片上传和访问。

等用户量和业务复杂度上来后，再拆分 AI 服务、文件服务或推荐服务。

## 10. 验收标准

- 用户能在 1 分钟内完成一条饮食记录。
- 用户能通过截图保存一条运动记录。
- 首页能同时展示今日和昨日的摄入、消耗、状态。
- 历史记录列表能展示至少最近 30 天数据。
- AI 结果可以被用户修改后保存。
- 每天能生成一条 AI 总结和一组明日建议。
- DeepSeek 返回失败时，用户仍可手动记录。
- 客户端主功能使用 Flutter 实现，不使用 WebView 承载核心页面。
- DeepSeek 只由 Java 服务端调用，客户端不保存 DeepSeek API Key。
- 基础信息页必须包含目标截止时间。
- 历史记录页点击某一天后，能进入单日详情页。
- 启动页点击“开始记录”后，能根据登录状态和资料完整度进入登录、基础信息或首页。
- 首页必须展示今日计划，昨日回顾样式必须与今日状态保持一致。
- 推荐页必须展示明日计划和 7 天计划日历。
- 推荐页必须提供“更新计划”按钮，支持基于昨天和历史情况重新生成计划。
- 单日详情页为只读页面，不提供编辑入口。
- 首页只保留一个版本，不再区分通用首页和老用户首页。
- 首页必须提供手动记饮食入口。
- 今日进度和昨日回顾必须支持点击进入对应日期的单日详情页。
- 手动记饮食必须有独立输入页面，支持餐次、食物名称、份量和热量。
- 底部导航必须使用图标加文字，并提供明确激活态，不使用纯文字圆点样式。
- 登录页必须包含手机号输入。
- 基础信息页必须包含昵称。
- 我的页必须展示昵称和脱敏手机号。

## 11. 原型范围

第一版 Figma 原型包含 11 个移动端页面：

1. 启动页。
2. 登录页。
3. 基础信息页。
4. 首页。
5. 饮食识别页。
6. 手动饮食记录页。
7. 运动识别页。
8. 推荐页。
9. 历史记录页。
10. 单日详情页。
11. 我的页。

视觉方向：
- 清爽、轻量、健康感。
- 避免复杂医疗感。
- 以白底、浅绿色、深墨色文字为主。
- 卡片圆角控制在 8px。
- 关键数据突出，辅助说明克制。
